# Trading RL Interview Notes

## 1. 这个项目解决什么问题？
把 OHLCV 时间序列建模为一个离散动作序列决策任务，让策略在 short / flat / long 三种动作之间切换，并在收益、交易成本和风险之间做权衡。

## 2. 状态、动作、奖励是怎么定义的？
- 状态：最近一段收益率窗口 + 当前持仓 + 短均线/长均线特征。
- 动作：0=short，1=flat，2=long。
- 奖励：`pnl - trade_cost - drawdown_penalty`。

## 3. 为什么奖励里要加交易成本和回撤惩罚？
如果只优化收益，策略容易频繁换仓或在高波动阶段承担过大风险。加入交易成本可以抑制无意义交易，加入回撤惩罚可以让策略对净值下行更敏感。

## 4. 你怎么评估策略，而不是只看训练 reward？
我把评估拆成两层：
- RL自身指标：累计收益、最大回撤、胜率、交易次数、动作分布。
- 传统基线比较：Buy-and-Hold、Moving Average。

## 5. 当前跑出了什么结果？
在 2026-04-18 这轮验证里：
- DQN 跑通了完整 train -> eval 链路。
- DQN 总收益约 -0.4044，优于 Buy-and-Hold 的 -0.4708，但弱于 Moving Average 的 -0.3642。
- PPO 在当前设定下塌缩成全程空仓，收益和回撤都接近 0。

## 6. PPO 空仓说明了什么？
这通常说明当前奖励尺度、探索方式或数据分布让“不交易”成为局部最优。这个结果不是失败信息，反而说明评估链路能识别策略塌缩，而不是只报告一个训练完成的模型。

## 7. 为什么现在还不能声称策略有效？
因为当前结果主要证明工程闭环已经成立，不代表研究结论稳定。限制包括：
- 这轮使用了 synthetic OHLCV fallback；
- 目前还是单 seed；
- 还没有 walk-forward 和样本外验证。

## 8. 项目里最能体现工程能力的部分是什么？
- 自定义 trading env，而不是只跑现成 CartPole。
- orchestrator 把训练和评估串成一条可复现实验链路。
- 评估会自动输出 JSON 报告，便于后续比较算法和参数。
- 数据下载失败时提供 fallback，保证实验能持续推进。

## 9. 如果继续做下一步，你会优先做什么？
1. 调 reward scaling 和 turnover penalty，减少 DQN 过少交易和 PPO 空仓。
2. 做 multi-seed 和 walk-forward。
3. 接稳定真实市场数据源，替换 synthetic fallback。
4. 增加更丰富的时序特征和报告可视化。
